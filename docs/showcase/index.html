为什么 HyDE 不加 Instruct 前缀？
Qwen3-Embedding 的 Instruct 前缀将 embedding 切换到 query-document 匹配空间。 HyDE 的核心是"以文搜文"——假设文档应与真实文档处于同一语义空间， 加前缀反而破坏 document-document 相似度的有效性。
双路融合策略：Max 而非 Average
两路 Bi-encoder 融合使用 max(score_rq, score_hyde)： 只要有一路命中，就能召回候选 chunk。 Average 会稀释高分路径的信号，Max 充分利用两路互补性，提升召回率。
两阶段的工程意义
Bi-encoder 预计算文档向量，支持毫秒级大规模检索。 Cross-encoder 联合编码不可预计算，但精度远高于 Bi-encoder。 Reranker 只处理 Top-50，单次延迟控制在秒级以内。

技术决策与踩坑

六个架构决策与真实踩坑

从架构取舍到真实数据暴露的问题，记录做了什么、没做什么，以及为什么。

架构决策
资料对齐质检为什么要在检索之前？
混合检索精排对"文件放对了文件夹"的 chunk 有 folder_boost 加权。 如果没有质检这一步，放错位置的文件反而会因为文件夹匹配被加权召回， 把错误信息带进初稿。

先质检、将 alignment_status 写入 chunk 缓存， 混合检索精排消费该字段对疑似错位 chunk 做反向降权，让信号闭环。
架构决策
为什么不在资料对齐质检阶段前置 HyDE？
曾讨论在资料对齐质检阶段也引入 HyDE 提升召回。 结论：HyDE 在资料对齐质检与混合检索精排两阶段的语义判断完全重叠， 且资料对齐质检的目标是"判断文件是否放错了位置"， 用假设文档反而会混淆"文件内容与指标的语义距离"和"文件归属"之间的边界， 导致误判率上升。明确不做。
架构决策
表格为什么不能用普通字符数分块？
按字符数切割会在表格中间断开，造成：
  • 表头丢失，chunk 无法独立理解
  • 同一数值行的列名上下文丢失
  • Reranker 对残缺表格打分不准
解决方案：表格与正文结构分离，独立按行数分块（max_rows=30），每块补充完整表头。
真实数据踩坑
客户资料按部门归档，folder_code 信号失效
系统设计预期客户按指标编码（GA1/EA1…）组织文件夹。 真实客户按部门组织（财务部/证券事务部/ESG-南通…）， 导致 folder_code 从路径提取全部为 None， FOLDER_BOOST_SCORE 等 4 个超参数无法从本次数据标定。

发现的解锁路径：大量文件名本身含编码（如 SD1、安全管理.pdf）， 从文件名提取 folder_code 可恢复信号，是下一步优化方向。
真实数据踩坑
HyDE hypothetical_doc 为 None 引发 TypeError
write_evaluation_sample() 中对 LLM 返回值使用了 r.get('hypothetical_doc', '')， 但 LLM 偶尔返回 null（Python 侧为 None）， None[:300] 直接抛 TypeError。

修复：r.get('hypothetical_doc') or ''， 显式处理 None 而非依赖 default 参数。 这是真实数据跑通混合检索精排时发现并修复的。
工程决策
retrieval_results.json 单文件存储而非双文件
早期方案将 Bi-encoder 结果和 Reranker 结果分别存两个文件。 问题：下游 LLM 撰稿需要判断用哪个文件；对比分析需要手动对齐。

改为单文件双套字段：每个 chunk 同时保留 score（reranker）、score_biencoder、 biencoder_rank，磁盘占用从 ~12MB 降至 ~7MB， 下游直接消费，无需感知是否启用了 reranker。
