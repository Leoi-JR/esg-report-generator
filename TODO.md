## P0 · 立即修复

### 1. 系统在证据不足时仍生成大段正文
- **类型**：功能优化  
- **核心依赖**：是  
- **支撑功能**：生成正文、结构约束  
- **预估工时**：2-3d  
- **问题描述**：当用户上传的资料仅覆盖部分章节时，系统未严格按资料匹配结果控制正文生成范围。对于缺乏有效支撑材料、匹配度低或无匹配结果的章节，系统应执行"跳过生成"或"输出待补提示"，但实际仍继续生成大段正文，导致报告出现明显脱离资料基础的泛化表述和臆造内容。  
- **影响判断**：当前最影响系统可用性和结果可信度，不修复则输出不可交付

#### 现状分析
- `generate_draft.py` L674 已有质量过滤：`max_score < DRAFT_SCORE_THRESHOLD(0.25)` 的章节标记 `skipped`，不调用 LLM
- `draft_user.txt` L27 已有 Prompt 指令：`如果资料不足以支撑完整章节，在末尾用 [待补充] 标注缺失内容`
- **根因**：Reranker 精排得分普遍高于 0.25 阈值，导致大量弱相关章节仍进入 LLM 生成，LLM 自由发挥产生臆造内容

#### 修复方案（无需分层，直接调阈值）
- 章节"应该写什么结构"由 Prompt 中的 `full_path` / `leaf_title` / `gloss` / `retrieval_query` 预先决定，与资料多寡无关
- 不需要 0.25~0.4 的"只生成框架"中间态——中间态反而给 LLM 在框架中填充编造内容的空间

#### 重难点
1. **确定合适的阈值**：需要分析 `retrieval_results.json` 中 119 个章节的 `max_score` 分布，找到"有效资料"与"弱相关噪声"的分界点
2. **加强 Prompt 约束力**：现有 Prompt 中"如果资料不足"是软性建议，LLM 不一定遵守；需改为更强硬的指令措辞
3. **后处理兜底（可选）**：LLM 输出后检查 `[来源X]` 引用密度，引用过少的章节自动追加 `[待补充]` 标记

## P1 · 优先开发

### 2. 正文自动识别和添加表格 + 列表格式（合并处理）
- **类型**：功能优化  
- **核心依赖**：是  
- **支撑功能**：正文表格 + 列表结构展示  
- **预估工时**：10-12d  
- **难度**：⭐⭐⭐⭐ 高  
- **需求描述**：
  - **表格**：从识别的材料中，将带表格结构的数据在正文中还原为表格形式；章节专属 prompt 有表格展示示例时，前端需支持表格结构渲染
  - **列表**：识别 `1、2、3`、`•`、`*` 等序号/项目符号，还原为列表结构展示

#### 现状分析
- ✅ `extractors.py` 已将表格拆分为独立 chunk，带 `is_table=True`、`table_html`、`table_markdown` 字段
- ✅ `table_summarizer` 已为表格 chunk 生成摘要用于 Embedding
- ✅ Tiptap StarterKit 已包含 BulletList / OrderedList（列表基础能力已有）
- ❌ 表格在 Reranker 精排中效果不如纯文本，召回率偏低
- ❌ 大表被 `split_table_by_rows()` 拆成多 chunk，检索时可能只召回碎片（丢失表头）
- ❌ Tiptap 未安装 Table 扩展，`contentToTiptapHTML()` 未处理表格 Markdown
- ❌ Word 导出不支持表格

#### 重难点
1. **表格召回率低**：表格 chunk 的 Embedding 语义质量天然不如自然语言段落，虽有 `table_summary` 增强，但 Reranker 精排看到的仍是 Markdown 表格原文，相关表格可能在 Top-10 之外就被淘汰。可能需要独立的表格召回策略（如按 `folder_code` 精确匹配补召回）
2. **表格跨 chunk 碎片化**：大表拆分后，检索只召回了包含关键词的中间片段，表头在另一个 chunk 中。需要通过 `parent_id` 或 `chunk_index` 相邻性自动拼接同源表格碎片
3. **LLM 对表格的使用策略**：LLM 倾向于把表格数据改写为段落叙述，但 ESG 报告中表格应保持原貌（数据不可改）。需要在 Prompt 中明确区分"引用表格（原样插入）"和"叙述表格数据"
4. **前端渲染与双向转换**：安装 Tiptap Table 扩展 + `contentToTiptapHTML` 处理 Markdown table + Word 导出适配（工程量明确，难度不高）
5. **列表格式归一化**：LLM 可能输出中文序号 `1、2、3` 而非标准 Markdown 列表，需要在 Prompt 约束或后处理中正则转换（相对简单）

### 3. 正文图片引用
- **类型**：功能优化  
- **核心依赖**：是  
- **支撑功能**：正文图片材料引用补充  
- **预估工时**：7-8d  
- **难度**：⭐⭐⭐ 中（路径清晰的工程问题，无需策略探索）  
- **需求描述**：
  - 支持正文中引用图片素材，包括**独立图片文件**和**文件嵌入式图片**两类
  - 需过滤嵌入式图片质量，排除 logo、签名、装饰图标等无效素材进入正文
  - 直接影响材料利用率和正文完整性，尤其是案例、环境、活动类章节

#### 现状分析
- ✅ VLM 分类（7 类）+ 描述文本 + OCR（扫描件/表格截图）全流程已完成
- ✅ 图片 chunk 以文本描述形式已进入 Embedding 索引
- ✅ `vlm_cache.json` 存储了分类和描述，质量过滤（≥100×100px 且 ≥5KB）已实现
- ❌ 原始图片未持久化存储，无可引用的 URL/路径
- ❌ Tiptap Image 扩展未安装
- ❌ SourcePanel 无图片预览能力
- ❌ Word 导出不支持嵌入图片

#### 重难点
1. **图片持久化与 URL 映射**：当前图片在提取后仅转为文本描述，原始图片未存储。需要设计存储方案（存到 `data/processed/images/` + 文件名映射）并在 chunk 中添加 `image_path` 元数据
2. **LLM Prompt 改造**：让 LLM 感知"哪些来源包含有价值图片"并在正文中标记（如 `[图片X]`），Prompt 模板需增加图片来源标注指令
3. **前端 Tiptap Image 扩展**：安装配置 `@tiptap/extension-image` + `contentToTiptapHTML` 中处理图片标记 + SourcePanel 中添加图片缩略图预览
4. **Word 导出嵌入图片**：`docx` 库支持图片插入，但需要处理图片尺寸适配和布局

### 4. 写作规则（由 P2 提升）
- **类型**：功能优化  
- **核心依赖**：是  
- **支撑功能**：生成正文  
- **预估工时**：8-10d  
- **难度**：⭐⭐ 低（Prompt 迭代为主，路径明确）  
- **整理产物**：Excel 表格（每个议题的正文结构、写作提醒、必备支撑要素）  
- **需求描述**：
  - 为每个议题（四要素和非四要素）制定正文结构规范、写作提醒、必备支撑要素
  - Prompt 模板与报告框架一一对应
  - 变更频率高（可按议题持续扩充）

#### 现状分析
- ✅ 已有通用 Prompt 模板（`draft_system.txt` + `draft_user.txt`），对所有 119 个章节使用相同指令
- ❌ 不同议题的写作结构、侧重点、必备要素差异大（如定量指标章节 vs 制度描述章节），通用 Prompt 无法区分

#### 重难点
1. **议题分类与 Prompt 映射**：需要建立"议题编码 → 写作规则"的映射表，让不同章节使用不同的补充 Prompt 指令
2. **写作规则内容产出**：规则本身需要 ESG 领域专业知识来撰写（需求方提供），开发侧主要是加载和注入机制
3. **效果验证**：Prompt 精细化后需要对比验证生成质量的提升，可能需要多轮迭代

- **类型**：功能开发  
- **核心依赖**：是  
- **支撑功能**：用户自行判断议题属性、生成议题散点图  
- **预估工时**：8d  
- **难度**：⭐⭐⭐ 中（前端交互开发为主）  
- **需求描述**：
  - 支持用户对议题的**影响重要性**和**财务影响重要性**进行打分
  - 生成议题双重重要性散点图
  - 构建报告模板时，已判定为双重重要性的议题不得低于对应象限阈值
- **调整原因**：独立产品模块，不影响核心生成链路；当前框架中无"四要素/非四要素"区分，需先改框架数据结构

#### 重难点
1. **打分 UI 交互设计**：需支持批量议题打分，交互体验要流畅
2. **ECharts 散点图集成**：双轴散点图 + 象限线 + 交互式标签
3. **与框架数据结构打通**：当前 `generate_retrieval_queries.py` 对所有叶节点一视同仁，需新增四要素/非四要素标记字段

### 6. 自定义报告框架功能（由 P1 调整）
- **类型**：功能开发  
- **核心依赖**：是  
- **支撑功能**：报告目录生成、章节映射、结构约束  
- **预估工时**：10-12d  
- **难度**：⭐⭐⭐⭐ 高  
- **整理产物**：Excel 表格（四要素与非四要素基础报告框架模板）  
- **需求描述**：
  - 重制四要素与非四要素的基础报告框架模板
  - 支持用户自定义报告框架结构
  - 是整套生成链路的入口
- **调整原因**：当前系统通过 `ESG报告框架.xlsx` 直接驱动全链路，解析逻辑已完善；"自定义"是产品化需求（服务多企业），应优先跑通单企业的生成质量

#### 重难点
1. **前端框架编辑器**：需要树形结构拖拽/增删改 UI，交互复杂度高
2. **全链路适配**：框架变更需级联到检索查询生成 → 混合检索 → 初稿撰写全流程
3. **内置模板管理**：CRUD + 导入导出 + 版本管理

### 7. 编制指引指标口径
- **类型**：功能开发  
- **核心依赖**：是  
- **支撑功能**：指标抽取、章节指标统计  
- **预估工时**：8d  
- **难度**：⭐⭐⭐ 中  
- **整理产物**：Excel 表格（按报告框架拆分交易所编制指引和机构指标，对应到报告框架）  
- **需求描述**：
  - 按报告框架结构拆分交易所编制指引（如上交所指引第13/4号）和机构指标（GRI、MSCI、WIND、CASS-ESG 等）
  - 将各指标对应到报告框架的具体章节
  - 后续可持续扩展，变更频率低

#### 重难点
1. **指标拆分粒度**：不同指引体系（GRI / CASS-ESG / 上交所）的指标颗粒度和命名方式不同，需统一到报告框架的章节层级
2. **多体系交叉映射**：同一章节可能对应多个体系的不同指标编号，需设计多对多映射结构
